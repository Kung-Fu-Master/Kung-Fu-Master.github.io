# 概念

MinIO 使用密钥管理系统（KMS）支持 SSE-S3。如果客户端请求 SSE-S3，或 启用了自动加密，则 MinIO 服务器会使用唯一的对象密钥对每个对象进行加密，该对象密钥受 KMS 管理的主密钥保护。

KMS 将 MinIO 作为面向应用程序的存储系统与安全密钥存储区分开，并且 可以由专门的安全团队进行管理。MinIO 通过我们的 KES project 支持常用的 KMS 实现，例如 Hashicorp Vault。 通过 KES，可以利用存储基础架构（MinIO 群集）水平扩展 KMS。 通常，MinIO-KMS 基础结构如下所示：

     ┌─────────┐         ┌────────────┐         ┌─────────┐  
     │  MinIO  ├─────────┤ KES Server ├─────────┤   KMS   │ 
     └─────────┘         └────────────┘         └─────────┘

当您将存储基础架构扩展到多个 MinIO 群集时，您的架构应如下所示：

    ┌────────────┐
    │ ┌──────────┴─┬─────╮          ┌────────────┐
    └─┤ ┌──────────┴─┬───┴──────────┤ ┌──────────┴─┬─────────────────╮
      └─┤ ┌──────────┴─┬─────┬──────┴─┤ KES Server ├─────────────────┤
        └─┤   MinIO    ├─────╯        └────────────┘            ┌────┴────┐
          └────────────┘                                        │   KMS   │
                                                                └─────────┘

请注意 ，所有 MinIO 群集均仅具有 其自己的 KES 实例的连接，而不能直接访问 Vault（作为一种可能的 KMS 实现）。 每个 KES 实例将处理 “其” MinIO 群集发出的所有加密 / 解密请求，从而使中央 KMS 实现不必处理 大量流量。相反，每个 KES 实例都将使用中央 KMS 实现作为安全密钥存储，并从中获取所需的主密钥。

该指南显示了如何使用 Hashicorp Vault 作为 KMS 实施来设置 MinIO-KMS 部署。 因此，它显示了如何设置和配置：

  • Vault 服务器作为中央密钥库。
  • 一个 KES 服务器实例，作为 MinIO 和保险柜之间的中间件。
  • MinIO 实例本身。

请注意, 为简便起见，本指南使用自签名证书。在生产部署中，应使用 由 “公共”（例如，让我们加密）或组织内部的 CA 颁发的 X.509 证书。

本指南说明如何在同一台计算机上设置三台不同的服务器：

  • Vault 服务器为 https://127.0.0.1:8200
  • KES 服务器为 https://127.0.0.1:7373
  • MinIO 在 K8s 上部署后 9000 映射到主机端口位 30007, 则服务器为 https://127.0.0.1:30007

# KMS(Vault)

# 为保险柜生成 TLS 证书

vault-crt.conf

[req]
distinguished_name = req_distinguished_name
x509_extensions = v3_req
prompt = no

[req_distinguished_name]
C = US
ST = state
L = location
O = organization
CN = localhost

[v3_req]
subjectAltName = @alt_names

[alt_names]
[alt_names]
IP.1 = 127.0.0.1
IP.2 = 10.239.140.73
DNS.1 = localhost
DNS.2 = master

生成证书

# 使用ECDSA生成密钥
openssl ecparam -genkey -name prime256v1 | openssl ec -out vault-tls.key
openssl req -new -x509 -nodes -days 30 -key vault-tls.key -sha384 -out vault-tls.crt -config vault-crt.conf

# 使用RSA生成密钥
openssl genrsa -out vault-tls.key 2048
openssl req -new -x509 -nodes -days 30 -key private.key -sha384 -out vault-tls.crt -config vault-crt.conf

查看密钥和证书

# 查看ecdsa生成的密钥
openssl ec -in vault-tls.key -text -noout
# 查看rsa生成的密钥
openssl rsa -in vault-tls.key -text -noout

(Optional)# 查看csr
(Optional)openssl req -in root-cert.csr -text -noout

# 查看证书
openssl x509 -in vault-tls.crt -text -noout

# 创建保管库配置文件

cat > vault-config.json <<EOF
{
   "api_addr": "https://127.0.0.1:8200",
   "backend": {
     "file": {
       "path": "vault/file"
     }
   },
  "default_lease_ttl": "168h",
  "max_lease_ttl": "720h",
  "listener": {
    "tcp": {
      "address": "0.0.0.0:8200",
      "tls_cert_file": "vault-tls.crt",
      "tls_key_file": "vault-tls.key",
      "tls_min_version": "tls12"
    }
  }
}
EOF

请注意，我们使用文件后端运行 Vault。为了获得高可用性，您可能需要使用其他 后端，例如 etcd 或 consul.

# 启动 Vault 服务器：

启动 vault server, 测试完终止后，下次直接运行此命令重启 vault server, 再用下面 三条 unseal 命令解封即可.

vault server -config vault-config.json

# 初始化和解封保管箱

切换终端，在单独的终端窗口中设置环境 VAULT_ADDR。保险柜服务器的变量:

export VAULT_ADDR='https://127.0.0.1:8200'
// 如果Vault使用自签名TLS 证书，则可能要运行如下命令, 当Vault提供 由计算机信任的CA颁发的TLS证书（例如，让我们加密）时，则无需运行此命令.
export VAULT_SKIP_VERIFY=true
// 初始化保险柜
vault operator init

保管箱将打印n（默认情况下为5）解封密钥共享，其中至少m（至少3） 为重新生成实际解封密钥才能解封保管库. 因此，请务必 记住它们。特别是，请将那些未密封的密钥共享放在安全且持久的位置
您应该看到类似于以下内容的输出：

# 启用保险柜的 K/V 后端.

加密主密钥（而不是对象加密密钥）将存储 在 Vault 中。因此，我们需要启用 Vault 的 K/V 后端。为此，请运行：

vault secrets enable kv

为 K/V 引擎 创建访问策略.
以下策略确定应用程序（即 KES 服务器）如​​何 与 Vault 交互.

cat > minio-kes-policy.hcl <<EOF
path "kv/minio/*" {
  capabilities = [ "create", "read", "delete" ]
}
EOF

观察在 kv/minio/* 的路径前缀 minio , 此前缀确保 KES 服务器只能在 minio/- 下进行读取，而只能在 - 下进行写入 some-app/ . 如何在 K/V 引擎上分隔域取决于您的基础结构 和安全要求。

然后，我们将政策上传到保险柜：

vault policy write minio-key-policy ./minio-kes-policy.hcl

# 启用 AppRole 身份验证

由于我们希望稍后将一个 / 多个 KES 服务器连接到 Vault，因此必须启用 AppRole 身份验证。为此，请运行：

vault auth enable approle

创建一个新的 AppRole ID 并将其绑定到策略.
创建一个新的 AppRole ID 并授予该 ID 特定的权限。 该应用程序（即 KES 服务器）将通过 AppRole 角色 ID 和机密 ID 向 Vault 进行身份验证，并且仅允许执行特定策略授予的操作.

# 我们首先为KES服务器创建一个新角色:
vault write auth/approle/role/kes-role token_num_uses=0  secret_id_num_uses=0  period=5m

# 然后，我们将策略绑定到角色：
vault write auth/approle/role/kes-role policies=minio-key-policy

# 最后，我们从Vault请求AppRole角色ID和秘密ID。
首先是角色ID：
vault read auth/approle/role/kes-role/role-id 
	fe46e329-bb70-2342-9fac-f3964a447d07

# 然后是秘密ID：
vault write -f auth/approle/role/kes-role/secret-id
	50646d24-5589-bba9-3f99-15ef2ae7e100

我们只对 secret_id 感兴趣，不包括 secret_id_accessor.

# KES

# 生成 KES 证书和私钥

此部署文档将Kes和MinIO共用相同的private.key和public.crt
上面私匙和证书名字不要改变，因为 MinIO 的私匙和证书必须是 private.key 和 public.crt,

kes-minio-crt.conf:

[req]
distinguished_name = req_distinguished_name
x509_extensions = v3_req
prompt = no

[req_distinguished_name]
C = US
ST = state
L = location
O = organization
CN = localhost

[v3_req]
subjectAltName = @alt_names

[alt_names]
IP.1 = 127.0.0.1
IP.2 = 10.239.140.73
DNS.1 = *.minio-hl-svc.minio.svc.cluster.local
DNS.2 = master
DNS.3 = localhost

连接到 KES 服务器（mTLS）时，每个用户或应用程序必须出示有效的 X.509 证书。 KES 服务器将接受 / 拒绝连接尝试，并根据证书应用策略。

因此，每个 MinIO 群集都需要一个 X.509 TLS 证书来进行客户端身份验证。可以通过运行以下命令来创建 MinIO（自签名）证书.

# 使用ECDSA生成密钥
openssl ecparam -genkey -name prime256v1 | openssl ec -out minio-pki/private.key
openssl req -new -x509 -nodes -days 730 -key minio-pki/private.key -sha384 -out minio-pki/public.crt -config kes-minio-crt.conf

# 使用RSA生成密钥
openssl genrsa -out private.key 2048
openssl req -new -x509 -nodes -days 730 -key minio-pki/private.key -out minio-pki/public.crt -config kes-minio-crt.conf

# KES和MinIO共用相同的证书和私钥, 因此拷贝一份MinIO密钥和证书给KES用:
cp minio-pki/private.key ./kes-server.key
cp minio-pki/public.crt ./kes-server.crt

注意，MinIO 是 subject name。 您可以为您的部署方案选择一个更合适的名称。此外，对于生产部署，我们 建议获取由 CA 颁发的用于客户端身份验证的 TLS 证书。

要获取 MinIO 的 X.509 证书的身份，请运行：

kes tool identity of minio-pki/public.crt
# 此命令可与任何（有效）X.509证书一起使用-无论如何创建它-并 产生类似于以下内容的输出：
  Identity: dd46485bedc9ad2909d2e8f9017216eec4413bc5c64b236d992f7ec19c843c5f

# 设置环境变量
export APP_IDENTITY=dd46485bedc9ad2909d2e8f9017216eec4413bc5c64b236d992f7ec19c843c5f

# 创建 KES 配置文件

创建 KES 配置文件并启动 KES 服务器.

cat > kes-server-config.yml <<EOF
address: 0.0.0.0:7373
root:    disabled  # We disable the root identity since we don't need it in this guide

tls:
  key:  kes-server.key
  cert: kes-server.crt

policy:
  my-app:
    paths:
    - /v1/key/create/minio-*
    - /v1/key/generate/minio-*
    - /v1/key/decrypt/minio-*
    identities:
    - ${APP_IDENTITY}

cache:
  expiry:
    any:    5m0s
    unused: 20s

keys:
  vault:
    endpoint: https://127.0.0.1:8200
    prefix: minio
    approle:
      id:     "fe46e329-bb70-2342-9fac-f3964a447d07" # Your AppRole ID: $ vault read auth/approle/role/kes-role/role-id
      secret: "50646d24-5589-bba9-3f99-15ef2ae7e100" # Your AppRole Secret ID: $ vault write -f auth/approle/role/kes-role/secret-id
      retry:  15s
    status:
      ping: 10s
    tls:
      ca: "vault-tls.crt" # Since we use self-signed certificates

EOF

请 identities 在 policy 部分中将的值更改为您的身份 minio-pki/public.crt, 或者在终端运行 export APP_IDENTITY=dd46485bedc9ad2909d2e8f9017216eec4413bc5c64b236d992f7ec19c843c5f
另外，插入您在保险柜设置过程中之前创建的 AppRole 角色 ID 和密码 ID。 您可以在此处找到包含所有可用参数的文档化配置文件.

最后，通过以下方式启动 KES 服务器：

kes server --mlock --config=kes-server-config.yml --auth=off

--auth=off is required since our root.cert and app.cert certificates are self-signed.
The --mlock option is currently only available on Linux.

有关 KES 访问控制模型和身份验证的更多信息:KES Concepts

# 创建一个新的主密钥

在继续进行 MinIO 设置之前，我们需要创建一个新的主密钥。因此，我们使用 MinIO 身份和 KES CLI。

在新的终端窗口中，通过以下方式成为 MinIO 身份：

export KES_CLIENT_CERT=minio-pki/public.crt
export KES_CLIENT_KEY=minio-pki/private.key

然后运行以下命令来创建主密钥：

# 设置访问vault变量:
export VAULT_SKIP_VERIFY=true
export VAULT_TOKEN=s.zaU4Gbcu0Wh46uj2V3VuUde0

# 创建主密钥:
kes key create -k minio-key-1

# 查看所有key-value文件夹
vault kv list kv/
  Keys
  ----
  minio/
  test01

# 查看某个文件夹下有哪些主密钥
vault kv list kv/minio
  Keys
  ----
  minio-key-1
  minio-key-2

# 获取某个主密钥数据信息
vault kv get kv/minio/minio-key-1
  ======= Data =======
  Key            Value
  ---            -----
  minio-key-1    {"bytes":"rrjxB3THaSzSGaJ2mUwWDjT/+Tvgb5XbECPi3P9rgBs="}

由于我们使用自签名证书，因此仅需要 -k 标志.
另外，请注意，基于服务器配置文件， 仅允许 MinIO 标识创建 / 使用以开头的主密钥 minio- . 因此，尝试创建密钥（例如） kes key create my-key-1 -k 将失败，并出现策略错误禁止的消息.

# 搭建本地 minio 服务

MinIO 服务器将需要知道 KES 服务器端点， 用于身份验证和授权的 mTLS 客户端证书以及默认的主密钥名称.

export MINIO_KMS_KES_ENDPOINT=https://10.239.140.73:7373
export MINIO_KMS_KES_CERT_FILE=minio-pki/public.crt
export MINIO_KMS_KES_KEY_FILE=minio-pki/private.key
export MINIO_KMS_KES_CA_PATH=kes-server.crt
export MINIO_KMS_KES_KEY_NAME=minio-key-1

由于我们使用自签名证书，因此只需要 ** MINIO_KMS_KES_CA_PATH **

然后启动 MinIO 服务器：

export MINIO_ACCESS_KEY=minio
export MINIO_SECRET_KEY=minio123
rm -rf /root/.mc
rm -rf /root/.minio
minio server --certs-dir ./minio-pki ./data
 * 指定上传minio server端证书存放路径文件夹为: ./minio-pki
 * minio存储的数据保存到data目录

切换终端，启动自动加密，如果在 K8S 上部署 MinIO server enable 了密钥和证书，则执行下面类似命令时候都要加上 --insecure , 信任自签的证书.

# 设置server服务地址
mc alias set myminio https://10.239.140.73:9000 minio minio123 --insecure

# 创建bucket
mc mb myminio/test01 --insecure

# 加密minio 已存在的某个 bucket
mc encrypt set sse-s3 myminio/test01/ --insecure
mc encrypt info myminio/test01/ --insecure
  Auto encryption 'sse-s3' is enabled

# 将本地文件synk.jpg上传到minio
mc cp synk.jpg myminio/test01/ --insecure

# 查看上传文件状态, 可以看到已自动加密
mc stat myminio/test01/synk.jpg --insecure

  Date      : 2021-03-15 11:54:36 CST
  Size      : 112 B
  ETag      : 2ced7fb62d324c643aeb21eccbe86eeb
  Type      : file
  Metadata  :
    Content-Type: application/x-sh
  Encrypted :
    X-Amz-Server-Side-Encryption: AES256

# 将上传的文件下载到本地
mc cp myminio/test01/synk.jpg 001.jpg --insecure
